# Security Notes

このドキュメントは、Maigent Agent Webを使う/開発する上でのセキュリティ上の注意点をまとめたものです。実際のコードとリポジトリ状態を確認した上で記載しています。

## 前提: このアプリの脅威モデル

Maigentはローカルで動くDjangoアプリですが、以下の理由で通常のローカルツールより注意が必要です。

- LLMが生成したPythonコードをDocker sandbox内で**実際に実行**する
- ユーザーが許可したローカルフォルダを**読み取り/書き込み**する
- ローカルファイルの内容やユーザーの質問文を、外部のLLM API・検索API(OpenAI互換API、Tavilyなど)へ送信する
- 複数のツール(RAG検索結果、web_search結果など)から取得した外部由来のテキストが、そのままLLMへの入力として使われる(プロンプトインジェクションの入口になり得る)

## 実装済みの安全対策(参考)

- sandboxはホストの許可フォルダを直接マウントしない。ファイル保存はstdout経由のtyped JSONをホスト側brokerが検証してから行う
- sandboxの`docker run`は既定で`--network none`(ネットワーク遮断)、`--memory`/`--pids-limit`/`--cpus`のリソース上限付き
- 書き込みは`file_write`機能フラグ + プロジェクトの書き出し先フォルダ配下のみに制限(`agent/file_broker.py`)
- 読み取りは`ProjectAccessPath`の許可リストのみ(`agent/access.py`の`is_path_allowed`は`Path.resolve()`でシンボリックリンクも解決した上でパス包含チェックを行う)
- ログ出力にはAPIキーらしき文字列をマスクする`RedactSecretsFilter`を適用(`agent/logging.py`)
- LLM生成コードはAST解析でローカルファイルアクセスや旧形式の成果物ペイロードを拒否するポリシーチェックあり(`agent/applications/sandbox.py`)

これらは前提として役立ちますが、以下の注意点は解消していません。

## ユーザーが気をつけるべき点

### 1. APIキーの取り扱い
`.maigent/config.yaml`(または`config.toml`)には各LLMプロバイダやTavilyのAPIキーを平文で記載します。このファイルは`.gitignore`済みですが、
- 誤って`git add -f`しない
- 設定ファイルを他人と共有・Slackなどに貼り付けない
- 定期的にキーをローテーションする
ことを推奨します。

### 2. sandboxは「隔離されているが無害ではない」
Docker sandboxはネットワーク遮断・リソース上限がありますが、**信頼できないコードを実行している**という前提は変わりません。
- sandbox機能は必要な時だけ有効化する
- 使用するDockerイメージ(`maigent-sandbox:py311`)は定期的に`docker build`し直し、ベースイメージやライブラリの脆弱性修正を取り込む
- Docker Desktop自体のアップデートも忘れずに

### 3. 許可フォルダ(access path)は最小限に
読み取り/書き込み許可を与えたフォルダの中身は、BM25検索やfile_batchによって**LLM APIへ送信される**可能性があります。
- ホームディレクトリ全体や機密情報を含むフォルダをまるごと登録しない
- プロジェクトごとに必要最小限のフォルダだけを許可する

### 4. RAG・web_searchは外部にデータが出る前提で使う
- RAGは許可フォルダの中身をLLM providerへ送信します。ローカルにあるからといって外部に出ないわけではありません。
- web_searchは質問文(検索クエリ)をTavily APIへ送信します。これも第三者サービスへの送信です。
- 機密性の高い作業では、これらのツールを無効化するか、許可フォルダを限定してください。

### 5. マルチエージェント実行時のAPI負荷
`multi_agent.enabled`が有効だと、複数workerが並列にLLM呼び出しを行います。ローカルLLM(Ollama/LM Studio)を使う場合は同時リクエストに耐えられるか確認してください。クラウドAPIの場合はコスト・レート制限にも注意してください。

### 6. このアプリには認証機能がありません
Django標準のログイン画面や権限チェックは組み込まれていません。`127.0.0.1`以外(社内LAN、外部ネットワークなど)にバインドして使う場合、**誰でもsandbox実行やファイル読み書きを行える状態**になります。ローカル(localhost)以外での起動は推奨しません。

## 今後懸念される点

- **プロンプトインジェクション**: RAGで読み込んだファイルやweb_searchの結果に悪意ある指示文が含まれていた場合、LLMがそれに従ってツール選択・ファイル書き込みを行ってしまう可能性があります。`dynamic_finalizer`や`sandbox`のポリシーチェックは部分的な緩和にしかなりません。外部由来のツール(file_batch、web_search)が増えるほどこのリスクは増加します。
- **公開デプロイへの拡張**: 将来的にlocalhost以外へ公開する場合、認証・認可・レート制限・HTTPS化が必須になります。現状のコードベースはそれらを一切前提にしていません。
- **sandboxコンテナの権限**: `docker/sandbox/Dockerfile`に`USER`指定が無く、コンテナ内はrootで実行されます。ネットワーク遮断・read-onlyマウントで緩和されていますが、コンテナエスケープ系の脆弱性が発見された場合の被害を抑えるため、非rootユーザー化や`--security-opt no-new-privileges`の追加を検討する価値があります。
- **ツール追加のたびのセキュリティレビュー**: 新しいツールを追加するたびに、アクセスパス制御・外部送信の有無・sandboxポリシーとの整合性を確認する運用を続けてください(`docs/functions.md`の「新しいツールを追加する方法」を参照)。
- **LLM呼び出しのループ・コスト暴走**: `dynamic_replanner`や`final_evaluation`のリトライは上限付きですが、設定を緩めすぎると意図しないAPIコスト増につながります。本番運用前に一通りの上限値を見直してください。

## このリポジトリ自体で修正すべき点(具体的な指摘)

以下はこのリポジトリの現状を確認した上での具体的な指摘です。優先度が高い順に並べています。

1. **`maigent_web/settings.py`の`DEBUG = True`と`SECRET_KEY`のハードコード、`ALLOWED_HOSTS`に`"*"`が含まれている**
   - `SECRET_KEY = "django-insecure-local-maigent-dev-key"`はDjangoの開発用ダミーキーがそのままリポジトリにコミットされています。ローカル専用である前提が崩れた場合(公開デプロイ、他人との共有)、セッション署名などが危殆化します。
   - `DEBUG = True`は例外発生時にスタックトレース・設定値・SQLクエリなどを画面に表示します。ローカル利用のみなら実害は小さいですが、`ALLOWED_HOSTS`に`"*"`が含まれているため、万一`0.0.0.0`などにバインドされた場合は誰でもこの詳細情報にアクセスできてしまいます。
   - 対応案: `SECRET_KEY`と`DEBUG`を環境変数から読み込む形にし、`ALLOWED_HOSTS`の`"*"`を外して`127.0.0.1`/`localhost`のみに限定する。本番相当の設定を使う場合は`.env`等で明示的に上書きできるようにする。

2. **`.codex_tmp/`がリポジトリにコミットされている(node_modulesへの壊れたシンボリックリンクを含む)**
   - `.codex_tmp/node_modules`は`/Users/ikedashinji/.cache/codex-runtimes/...`という**開発者個人のマシン上の絶対パス**を指すシンボリックリンクとしてコミットされています。他の環境では解決できず、開発者のローカルユーザー名・ディレクトリ構成が公開リポジトリ上に残ってしまっています。
   - `.codex_tmp/`自体もxlsx調査用のスクラッチスクリプト(`dump_workbook.mjs`など)で、アプリ本体とは無関係です。
   - 対応案: `.codex_tmp/`をリポジトリから削除し、`.gitignore`に追加する。

3. **`document.xlsx`が用途不明なままコミットされている**
   - `document.xlsx`は本アプリのドキュメント体系(`docs/`)と無関係な単体ファイルで、中身が確認されないままリポジトリ(公開リポジトリ)に残っています。個人情報や意図せず含まれた機密データが無いか、内容を確認することを推奨します。
   - (`codex_report_no_links.md`は本リポジトリと無関係な別ツールのメモだったため削除済み)

4. **Docker sandboxコンテナがrootユーザーで実行される**
   - `docker/sandbox/Dockerfile`に`USER`指定が無いため、コンテナ内プロセスはrootで動作します。現状は`--network none`とread-onlyマウントで緩和されていますが、非rootユーザーの追加(`USER appuser`など)と`--security-opt no-new-privileges`の付与を推奨します。

5. **認証機構が存在しない**
   - `django.contrib.auth`はINSTALLED_APPSに含まれ`/admin/`もルーティングされていますが、ログイン必須化やユーザー管理は一切実装されていません。localhost専用ツールとして使う分には許容範囲ですが、その前提をREADMEなどに明記し、「ネットワーク公開しないこと」を利用者に明確に伝えるべきです。

6. **sandboxリソース上限はメモリ/PID/CPUのみ**
   - 今回`--memory`/`--pids-limit`/`--cpus`を追加しましたが、ディスクI/Oやコンテナのファイルシステム書き込み(`/work`以外の一時領域)への制限はまだありません。将来的に`--read-only`ルートファイルシステム化や`tmpfs`サイズ制限の追加を検討してください。
